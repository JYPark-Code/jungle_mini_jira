1. 왜 세션을 선택했는가?
이 프로젝트에는 세션이 맞습니다. 이유는 명확합니다.
Jinja2 SSR 기반이라 브라우저가 직접 HTML을 받는 구조인데, 쿠키는 브라우저가 자동으로 처리해주니 jQuery/AJAX에서 추가 작업이 없습니다. 
JWT를 쓰면 매 페이지 요청마다 JS로 토큰을 붙여야 하는데, SSR과 맞지 않습니다.
서버도 EC2 한 대이니 세션 공유 문제가 아예 없고, 모바일 앱이나 외부 API 클라이언트도 없습니다. 
AI 코드 생성 금지 조건에서 3~4일 안에 끝내야 하는데, 세션이 구현량이 압도적으로 적습니다.
설계 문서에서도 세션을 추천하고 있었고, 이미 app.py 뼈대에 session 기반으로 잡아두었으니 그대로 가면 됩니다.
